# Flappy Bird DQN Auto-Search 使用说明书 (V2)

## 1. 项目作用

本项目用于对 Flappy Bird 的 DQN 智能体进行自动超参数搜索。

V2 在 V1 基础上扩展了：
- 模块化代码结构（从单文件拆分为 7 个职责模块）
- n-step 扩展（支持 1/3/5 步回报）
- Prioritized Experience Replay（PER）
- 奖励比例搜索（死亡惩罚比例、存活奖励、缩放、裁剪）
- checkpoint 正式资产化（sha256 校验、版本追踪）
- 实验报告自动生成

当前主程序入口：
- `main.py`（V2 主入口）
- `flappy_bird_dqn_auto_search.py`（向后兼容入口，等同于 `main.py`）

## 6. 启动方式

### 6.1 只跑一次 baseline

用于快速确认当前默认配置是否能正常训练和评估：

```powershell
python main.py --baseline-only --mode debug
```

### 6.2 调试模式搜索

适合第一次试跑，规模小、速度快：

```powershell
python main.py --mode debug --max-trials 5
```

### 6.3 正常模式搜索

用于正式的中等规模搜索：

```powershell
python main.py --mode normal --max-trials 100
```

### 6.4 深度模式搜索

用于更大帧预算、更重评估的搜索：

```powershell
python main.py --mode deep --max-trials 100
```

### 6.5 V2 增强 — 启用 n-step

```powershell
python main.py --mode debug --max-trials 10 --n-step 3
```

### 6.6 V2 增强 — 启用 PER

```powershell
python main.py --mode debug --max-trials 10 --priority --per-alpha 0.6 --per-beta-start 0.4
```

### 6.7 V2 增强 — 启用奖励搜索

```powershell
python main.py --mode debug --max-trials 10 --death-ratio 10 --reward-scale 0.1
```

### 6.8 V2 增强 — 生成实验报告

```powershell
python main.py --mode debug --max-trials 10 --report
```

### 6.9 渲染历史最佳 agent 的动态演示

用于直接加载历史最佳 trial 的 checkpoint，并用 `pygame` 播放 greedy demo：

```powershell
python main.py --render --history search_history.jsonl
```

指定演示局数和渲染帧率：

```powershell
python main.py --render --history search_history.jsonl --render-episodes 3 --render-fps 60
```

说明：
- `--render` 不会启动 Optuna 搜索
- 它会直接从 `--history` 指向的历史文件中找到当前最佳 trial
- 然后读取该 trial 对应的 `checkpoint_path` 并渲染画面
- 如果没有最佳 trial，或没有 checkpoint，程序会直接报错退出

## 7. 命令行参数说明与命令组成规则

### 7.1 一条启动命令的通用模板

本项目的大多数启动命令都可以按下面这个模板来理解：

```powershell
python main.py [启动类型] [运行模式] [试验预算] [输出文件] [可选覆盖参数] [附加功能]
```

最常见的搜索命令模板是：

```powershell
python main.py --mode <debug|normal|deep> --max-trials <N> [其他可选参数]
```

例如：

```powershell
python main.py --mode normal --max-trials 30 --history normal_run1.jsonl --study-db normal_run1.db
```

这条命令不是一个“特殊写法”，只是把几个常用组成部分拼在一起：
- `python`：用当前 Python 解释器启动程序
- `main.py`：主入口脚本
- `--mode normal`：选择 `normal` 模式
- `--max-trials 30`：这次最多跑 30 个 Optuna trial
- `--history normal_run1.jsonl`：把历史记录写到 `normal_run1.jsonl`
- `--study-db normal_run1.db`：把 Optuna study 写到 `normal_run1.db`

### 7.2 先判断你要启动哪一类任务

在拼命令之前，先判断你要做的是哪一类事情：

1. `baseline-only`
只跑一次基线，不做搜索。

模板：

```powershell
python main.py --baseline-only --mode <debug|normal|deep> [其他可选参数]
```

2. `search`
正式搜索超参数。这是最常见的类型。

模板：

```powershell
python main.py --mode <debug|normal|deep> --max-trials <N> [其他可选参数]
```

3. `render`
不训练、不搜索，直接把历史最佳 agent 渲染出来。

模板：

```powershell
python main.py --render --history <history.jsonl> [--render-episodes N] [--render-fps N]
```

说明：
- `--baseline-only` 和 `--render` 都属于“高层模式开关”。
- 这两种情况下，程序行为和普通搜索不同。
- 如果你既不写 `--baseline-only`，也不写 `--render`，程序默认就是“搜索模式”。

### 7.3 搜索命令通常由哪几块组成

一条完整的搜索命令，通常由下面 6 块组成：

1. 固定入口

```powershell
python main.py
```

2. 运行模式

```powershell
--mode debug
```

或：

```powershell
--mode normal
```

或：

```powershell
--mode deep
```

3. 搜索预算

```powershell
--max-trials 30
```

可选地再加：

```powershell
--max-trial-frames 1000000
```

这表示：
- `--max-trials` 控制 trial 数量
- `--max-trial-frames` 控制单个 trial 最多训练多少帧

4. 输出文件

```powershell
--history normal_run1.jsonl --study-db normal_run1.db
```

如果还想把 checkpoint 单独放到一个目录：

```powershell
--checkpoint-dir normal_run1_checkpoints
```

5. 可选覆盖参数

例如：

```powershell
--n-step 3
```

或：

```powershell
--priority --per-alpha 0.6 --per-beta-start 0.4
```

或：

```powershell
--death-ratio 10 --reward-scale 0.1
```

6. 附加功能

例如：

```powershell
--report
```

表示搜索结束后自动生成报告。

### 7.4 推荐的拼接顺序

虽然命令行参数在大多数情况下不强制要求固定顺序，但建议你始终按下面顺序写，最不容易乱：

```powershell
python main.py
--baseline-only / --render / 默认搜索
--mode ...
--max-trials ...
--max-trial-frames ...
--history ...
--study-db ...
--checkpoint-dir ...
--n-step / --priority / reward 参数
--report
```

也就是说，推荐顺序是：
- 先写“这次要干什么”
- 再写“跑多大规模”
- 再写“结果保存到哪”
- 最后写“附加覆盖参数”

### 7.5 逐段拆解示例

示例 1：

```powershell
python main.py --mode debug --max-trials 5
```

含义：
- `python main.py`：运行主程序
- `--mode debug`：用 debug 预设
- `--max-trials 5`：只跑 5 个 trial

适用场景：
- 首次试跑
- 检查流程是否正常
- 不追求成功率，只看系统是否工作

示例 2：

```powershell
python main.py --mode normal --max-trials 30 --history normal_run1.jsonl --study-db normal_run1.db
```

含义：
- `--mode normal`：使用 normal 预算
- `--max-trials 30`：本次最多跑 30 个 trial
- `--history normal_run1.jsonl`：历史记录写到 `normal_run1.jsonl`
- `--study-db normal_run1.db`：study 数据库写到 `normal_run1.db`

适用场景：
- 第 1 轮正式搜索
- 希望把这次实验和默认文件分开保存
- 后续方便继续续跑这同一轮实验

示例 3：

```powershell
python main.py --mode normal --max-trials 30 --history normal_run1.jsonl --study-db normal_run1.db --priority --per-alpha 0.6 --per-beta-start 0.4 --report
```

含义：
- 前半段定义一轮 normal 搜索
- `--priority --per-alpha 0.6 --per-beta-start 0.4`：强制启用 PER 并覆盖部分参数
- `--report`：结束后生成报告

### 7.6 输出文件名应该怎么起

如果你不写 `--history` 和 `--study-db`，程序会使用默认文件：
- `search_history.jsonl`
- `optuna_study.db`

但更推荐按“实验轮次”自己命名，例如：

```powershell
--history normal_run1.jsonl --study-db normal_run1.db
```

或：

```powershell
--history deep_run1.jsonl --study-db deep_run1.db
```

命名建议：
- `debug_run1.jsonl` / `debug_run1.db`
- `normal_run1.jsonl` / `normal_run1.db`
- `deep_run1.jsonl` / `deep_run1.db`
- `normal_per_run1.jsonl` / `normal_per_run1.db`

这样你一眼就知道：
- 这是什么模式
- 是第几轮实验
- 是否是某个特殊配置

### 7.7 哪些参数建议成对出现

1. `--history` 和 `--study-db`

建议一起写。因为：
- `history` 负责记录结果
- `study-db` 负责保存 Optuna 搜索状态

2. `--history`、`--study-db`、`--checkpoint-dir`

如果你想把一次实验完全隔离保存，建议三者一起指定。

例如：

```powershell
python main.py --mode normal --max-trials 30 --history run2.jsonl --study-db run2.db --checkpoint-dir run2_checkpoints
```

3. `--priority`、`--per-alpha`、`--per-beta-start`

如果你要显式控制 PER，通常一起写更清楚。

4. `--render` 和 `--history`

渲染模式下，通常应该显式写 `--history`，避免误读错历史文件。

### 基础参数（V1 兼容）

| 参数 | 默认值 | 说明 |
|---|---|---|
| `--mode` | normal | debug / normal / deep |
| `--max-trials` | 100 | Optuna 最多执行的 trial 数 |
| `--max-trial-frames` | 按 mode | 覆盖单 trial 帧预算 |
| `--history` | search_history.jsonl | JSONL 历史文件路径 |
| `--study-db` | optuna_study.db | Optuna SQLite 数据库路径 |
| `--checkpoint-dir` | checkpoints | 模型 checkpoint 保存目录 |
| `--n-startup-trials` | 30 | TPE 冷启动随机 trial 数 |
| `--baseline-only` | - | 只跑 baseline，不搜索 |
| `--render` | - | 渲染历史最佳 agent |
| `--render-episodes` | 1 | 渲染局数 |
| `--render-fps` | 60 | 渲染帧率 |

### V2 新增参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| `--n-step` | None | 覆盖 n_step（1/3/5），不传则由搜索空间决定 |
| `--priority` | False | 启用 PER |
| `--per-alpha` | 0.6 | PER alpha 参数 |
| `--per-beta-start` | 0.4 | PER beta 起始值 |
| `--per-beta-train-updates` | 50000 | PER beta 按 train_updates 推进长度 |
| `--death-ratio` | None | 死亡惩罚比例（5~100） |
| `--alive-ratio` | None | 存活奖励比例（0~0.01） |
| `--reward-scale` | None | 训练 reward 缩放系数 |
| `--reward-clip` | None | 训练 reward 裁剪阈值 |
| `--report` | False | 搜索结束后生成实验报告 |

补充说明：
- 上述 V2 参数属于 CLI 覆盖参数。传入后会同时作用于 baseline 和每个 trial 的最终训练配置。
- 未传入的字段仍由 baseline 默认值或 Optuna 搜索空间决定。
- `--priority` 只有“强制开启”这一种覆盖语义；当前没有 `--no-priority` 开关。
- Optuna 终端里打印的 `parameters: {...}` 是 trial 的原始采样参数，不一定等于最终生效配置。
- 以 `search_history.jsonl` 中每条记录的 `config` 字段为最终真实配置。

## 10. 输出文件说明

### 10.1 `search_history.jsonl`

用途：
- 保存每次 baseline / trial / recheck 的结果记录

特点：
- 一行一个 JSON 对象
- 适合后处理、统计、导入 Python 分析

### 10.2 `optuna_study.db`

用途：
- 保存 Optuna 的搜索状态

特点：
- 用于断点续跑
- 下次运行时只要继续使用同一个文件，就会接着已有 study 继续搜索

### 10.3 `checkpoints/`

用途：
- 保存每次成功 trial 的模型权重（V2 正式资产化）

特点：
- 每个 checkpoint 包含 q_net 权重、target_net 权重、config、版本字段、sha256 校验
- `--render` 模式会读取这里的 `.pt` 文件

### 10.4 V2 报告产物

使用 `--report` 参数后生成：

| 文件 | 说明 |
|---|---|
| `summary_report.md` | 人类可读的实验报告（10 节：概览、最优、Top-K、状态分布、n-step 对比、PER 对比、奖励对比、checkpoint、版本、建议） |
| `topk_summary.json` | Top-K trial 结构化摘要 |
| `recheck_summary.json` | 最新 recheck 结果汇总 |
| `experiment_manifest.json` | 实验元信息（版本、trial 数、路径等） |

## 12. 断点续跑说明

如果搜索被中断，只要保留原来的：
- `--study-db`
- `--history`
- `--checkpoint-dir`

再次执行同类命令即可继续。

例如第一次运行：

```powershell
python main.py --mode debug --max-trials 20 --history run1.jsonl --study-db run1.db
```

中断后再次运行：

```powershell
python main.py --mode debug --max-trials 20 --history run1.jsonl --study-db run1.db
```

程序会读取已有 study，继续剩余 trial，而不是从头开始。

说明：
- 断点续跑依赖 `study-db` 和 `history`
- `--render` 依赖历史记录里的 `checkpoint_path` 能对应到真实 checkpoint 文件
- 如果你清空了 checkpoint 目录，但还保留着历史文件，那么 `--render` 会报错

## 13. Ctrl+C 中断行为

运行过程中按一次 `Ctrl+C`：

- 程序不会立刻粗暴退出
- 会等待当前 trial 结束
- 当前结果会尽量正常写入历史文件
- 然后输出 summary 并退出

如果再次 `Ctrl+C`，程序会强制退出。

## 15. `search_history.jsonl` 字段说明

`search_history.jsonl` 中实际主要有两类记录：

- `record_type = "trial"`：训练记录。baseline 也属于这一类，通过 `source = "baseline"` 区分。
- `record_type = "recheck"`：Top-K 复核记录。

### 15.1 trial 记录字段

常见字段说明如下：

| 字段 | 说明 |
|---|---|
| `trial_id` | trial 编号（baseline 通常为 -1） |
| `record_type` | 记录类型。正式训练记录固定为 `trial` |
| `timestamp` | 记录写入时间 |
| `config` | 完整超参数配置（含 V2 新增字段） |
| `source` | 记录来源（baseline / startup / tpe / retry_best / recheck） |
| `seed` | 随机种子 |
| `status` | 结果状态（success / failure / pruned / crashed） |
| `objective` | Optuna 优化目标值（越小越好） |
| `train_raw_env_frames` | 训练阶段消耗的环境帧数（不含评估） |
| `eval_raw_env_frames` | 评估阶段消耗的环境帧数 |
| `total_raw_env_frames` | 总环境帧数（train + eval） |
| `decision_steps` | DQN 实际做动作决策的步数 |
| `episodes` | 训练过程中完成的 episode 数 |
| `best_train_score` | 训练过程中的最高游戏分数 |
| `best_eval_score` | 评估中位数最高分（保留兼容） |
| `best_eval_median_score` | 评估中位数最高分（推荐使用） |
| `final_eval_scores` | 最终评估每个 episode 分数列表 |
| `success_rate_1000` | 分数 >=1000 的 episode 比例 |

| `mean_score` | 最终评估分数平均值 |
| `failure_reason` | 失败原因（成功时为空） |

| `duration_sec` | 本次 trial 耗时（秒） |

| `checkpoint_path` | checkpoint 文件路径 |

## 16. 常见排错

### 16.1 提示 `unrecognized arguments`

先执行：

```powershell
python main.py -h
```

当前版本的帮助输出中应该能看到这些 V2 参数：
- `--n-step`
- `--priority`
- `--per-alpha`
- `--per-beta-start`
- `--per-beta-train-updates`
- `--death-ratio`
- `--alive-ratio`
- `--reward-scale`
- `--reward-clip`

如果帮助输出里没有这些参数，通常说明你当前运行的不是最新合并后的 `main.py`。

### 16.2 Optuna 日志里的参数看起来没被覆盖

例如你传了：

```powershell
python main.py --mode debug --max-trials 10 --n-step 3
```

Optuna 自己打印的 `Trial ... parameters: {...}` 仍可能显示采样得到的 `n_step` 值。
这不是最终训练配置。

请以 `search_history.jsonl` 中对应记录的：
- `config.n_step`
- `config.priority`
- `config.per_alpha`
- `config.per_beta_start`

等字段为准，它们才是实际传给训练过程的最终配置。
