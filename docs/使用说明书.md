# Flappy Bird DQN Auto-Search 使用说明书（V3）

## 1. 项目目标

本项目用于在固定 Flappy Bird 标准环境中，自动搜索 DQN 训练配置，目标是在尽可能少的真实环境帧数内，稳定达到 `1000` 分。

当前主入口：

- `main.py`
- `flappy_bird_dqn_auto_search.py`
- `auto_workflow.py`

其中 `flappy_bird_dqn_auto_search.py` 只是兼容旧入口，实际建议始终使用 `main.py`。

`auto_workflow.py` 是自动工作流入口，目前已完成到 `V0.3` 的主链闭环：

- 自动创建运行目录
- 写入 `workflow_state.json`
- 写入 `stage_status.json`
- 自动执行 `baseline_check`
- 自动执行 `protocol_matrix`
- 自动执行 `structure_matrix`
- 自动生成 `selection_decision.json`
- 自动生成 `focused_search_space.json`
- 自动生成 `best_config.json`
- 自动生成 `best_config_meta.json`
- 自动启动一轮 `focused_search_round`
- focused search 失败后自动切到 `warmstart_search_round`
- warmstart 失败后自动切到 `population_search_round`
- population 仍失败时自动输出 `blocked_report.json`
- 出现成功 trial 后，根据 profile 自动进入 `recheck_topk`、`final_confirm` 或 `report`
- `cpu_quick` 默认直接跳过 `recheck_topk` 和 `final_confirm`
- `cpu_normal` / `cpu_deep` 在 `recheck_topk` 通过后进入 `final_confirm`
- `final_confirm` 完成后自动输出最终 `report/final_report.json`
- 已完成时重复启动会跳过 baseline，不会重复污染同一轮 history

它现在仍然偏简化的部分主要是：

- 更复杂的多轮自动收窄策略
- 更细的 temporarily-disabled / permanently-eliminated 候选管理
- 更完整的可视化与人工诊断报告

## 2. 当前 V3 CLI 能做什么

当前 CLI 已支持以下任务类型：

- 单次 `baseline` 验证
- 标准 TPE 搜索：`tpe_fresh`
- 基于历史优质配置的预热搜索：`warmstart_tpe`
- 自进化人口搜索：`population_async`
- 从 V3 snapshot 继续同一 trial：`--resume`
- 对历史 Top-K 配置做复核：`--recheck-topk`
- 对单个配置做最终 5-seed 验收：`--final-confirm`
- 运行预定义实验矩阵：`--matrix`
- 使用历史最佳 checkpoint 做 `pygame` 动态渲染演示：`--render`
- 搜索结束后生成报告：`--report`
- 启动自动工作流的最小 `baseline_check`：`auto_workflow.py`

## 2.2 自动工作流最小启动方式

当前 `auto_workflow.py` 已落地到 `V0.3` 的主链闭环，所以它的用途是：

- 先验证自动工作流运行目录、状态落盘、baseline / protocol / structure / focused_search 阶段执行和幂等恢复
- 不是替代完整 V3 搜索 CLI

推荐命令：

```powershell
python auto_workflow.py --goal stable_1000 --profile cpu_quick --run-dir runs/auto_run1
```

profile 含义：

- `cpu_quick`
  用途是快速验证自动工作流主链、状态落盘和候选筛选。当前默认 `recheck_topk=0`，且不做 `final_confirm`。
- `cpu_normal`
  用途是标准自动工作流验收。当前默认 `recheck_topk=3`，且会继续执行 `final_confirm`。
- `cpu_deep`
  用途是更重预算的完整自动搜索。当前默认 `recheck_topk=5`，且会继续执行 `final_confirm`。

这条命令执行并多次继续恢复后，当前应得到的核心产物是：

- `runs/auto_run1/workflow_state.json`
- `runs/auto_run1/stage_status.json`
- `runs/auto_run1/baseline/history.jsonl`
- `runs/auto_run1/baseline/baseline_summary.json`
- `runs/auto_run1/protocol_matrix/history.jsonl`
- `runs/auto_run1/protocol_matrix/summary.json`
- `runs/auto_run1/structure_matrix/history.jsonl`
- `runs/auto_run1/structure_matrix/summary.json`
- `runs/auto_run1/selection_decision.json`
- `runs/auto_run1/focused_search_space.json`
- `runs/auto_run1/best_config.json`
- `runs/auto_run1/best_config_meta.json`
- `runs/auto_run1/focused_search_round_1/history.jsonl`
- `runs/auto_run1/workflow_summary.json`
- `runs/auto_run1/report/final_report.json`

条件产物：

- `runs/auto_run1/recheck/recheck_summary.json`
  只会在 `cpu_normal` 或 `cpu_deep` 且工作流进入 `recheck_topk` 时生成。`cpu_quick` 默认不会生成这个文件。
- `runs/auto_run1/final_confirm/final_confirm_summary.json`
  只会在 `cpu_normal` 或 `cpu_deep` 且 `recheck_topk` 通过后生成。`cpu_quick` 默认不会生成这个文件。

当前自动工作流执行完第一轮 focused search 后，`workflow_state.json` 里的 `current_stage` 会根据结果进入：

- `report`
  说明在 `cpu_quick` 下，focused search 已出现成功 trial，工作流直接输出报告，后续是否继续做 recheck/final confirm 由人工决定。
- `recheck_topk`
  说明在 `cpu_normal` 或 `cpu_deep` 下，focused search 已出现成功 trial
- `warmstart_search_round`
  说明 focused search 尚未出现成功 trial，后续应进入 V0.2 的自动升级链

继续运行后，还可能进入：

- `population_search_round`
  说明 warmstart 仍未产生成功 trial，工作流已继续升级
- `blocked`
  说明 population 仍未产生成功 trial，当前搜索空间已判定为暂时走不通

若搜索链中出现成功 trial，后续还会继续进入：

- `recheck_topk`
  说明工作流正在对当前最佳候选做多 seed 复核
- `final_confirm`
  说明复核已通过，且当前 profile 允许做最终确认
- `report`
  说明工作流正在导出最终报告。对于 `cpu_quick`，它可能直接接在成功搜索阶段之后。
- `complete`
  说明当前自动工作流主链已执行完成

当进入 `blocked` 时，应额外看到：

- `runs/auto_run1/blocked_report.json`

其中至少会记录：

- 当前最佳配置路径
- 当前最佳配置内容
- 当前保留/淘汰候选
- 最近两轮搜索阶段的提升幅度
- 最近一轮失败原因统计
- 建议下一步动作

其中关键字段通常包括：

- `current_best_config_path`
- `current_best_config`
- `recent_round_improvements`
- `failure_reason_counter`
- `recommended_next_actions`

常见 `blocked_reason`：

- `search_space_exhausted`
  说明已跑到当前自动升级链尾部，仍没有可用成功候选。
- `no_viable_candidate`
  说明进入了 `recheck_topk`，但复核后没有候选通过。
- `repeated_runtime_failure`
  说明同一阶段连续两次运行期异常，工作流已停止并要求人工处理。

`report/final_report.json` 中则会额外固定输出：

- `best_config`
- `best_config_meta`
- `best_config_explanation`

其中 `best_config_explanation` 用来回答三件事：

- 为什么当前配置被视为最佳
- 它现在是否已经 `confirmed`
- 如果还没 `confirmed`，卡在 `recheck_topk`、`final_confirm` 还是某个 `blocked_reason`

## 2.1 这些指令分别是什么意思

这一节专门解释“命令背后的概念”和“它在整个实验流程里是拿来做什么的”。

### 2.1.1 `baseline-only` 是什么

`baseline-only` 的意思是：

- 不做超参搜索
- 只用一套固定默认参数跑一次完整训练

它的用途是：

- 验证当前代码、环境、依赖、checkpoint 写入是否正常
- 给后续搜索提供一条基线成绩
- 快速判断“系统能不能跑起来”

它不适合用来回答“这个项目现在能不能稳定到 1000 分”，因为它只跑一次固定配置。

推荐命令：

```powershell
python main.py --baseline-only --mode debug --history baseline_check.jsonl --checkpoint-dir baseline_check_ckpts
```

### 2.1.2 `tpe_fresh` 是什么

`tpe_fresh` 是默认的超参搜索方式。

它的概念是：

- 用 Optuna 的 TPE 方法做超参数优化
- 从空白状态开始搜索
- 每个 trial 都是从头训练一个新配置

它的用途是：

- 做第一轮正式搜索
- 在还没有太多历史经验时探索参数空间
- 为后续 warmstart、population、自定义矩阵提供历史数据

如果你不知道先跑什么，通常先跑它。

推荐命令：

```powershell
python main.py --mode normal --max-trials 30 --history tpe_run1.jsonl --study-db tpe_run1.db --checkpoint-dir tpe_run1_ckpts
```

### 2.1.3 `warmstart_tpe` 是什么

`warmstart_tpe` 的意思是“带历史经验的 TPE 搜索”。

它的概念是：

- 搜索器仍然是 TPE
- 但不是完全冷启动
- 会把历史里表现较好的配置先喂给搜索器，作为热启动样本

它的用途是：

- 在你已经跑过多轮搜索后继续提效
- 减少完全随机冷启动带来的浪费
- 利用旧实验结果继续推进搜索

它适合“已经有历史数据，但还想继续用 Optuna/TPE 框架”的阶段。

推荐命令：

```powershell
python main.py --mode normal --search-strategy warmstart_tpe --max-trials 30 --history warmstart_run1.jsonl --study-db warmstart_run1.db --checkpoint-dir warmstart_run1_ckpts
```

### 2.1.4 `population_async` 是什么

`population_async` 可以理解成“自进化搜索”。

它的概念是：

- 同时维护一小群候选配置，称为 population
- 每个 worker 先训练一小段，再评估
- 表现差的 worker 会被表现好的 worker 继承/替换
- 继承时不是完全重头训练，而是接着父 snapshot 继续训练
- 继承后会对部分参数做小幅变异，这就是 exploit/explore

它的用途是：

- 做 V3 的自进化超参优化
- 把“保留强者、淘汰弱者、边训练边微调参数”落到自动流程里
- 比纯 fresh trial 更强调“训练中继承”和“在线进化”

它特别适合：

- 你已经有一些不错的历史配置
- 想让系统自己沿着已有好方向继续爬坡
- 想测试“继承训练 + 小步变异”是否比从头搜索更有效

需要注意：

- 它不是传统意义上的真正多进程大规模并行平台
- 当前实现更接近“单机轮转式人口进化搜索”
- 它的核心价值是搜索机制，不是吞吐量极限

推荐命令：

```powershell
python main.py --mode normal --search-strategy population_async --population-size 2 --population-eval-interval 20000 --population-exploit-interval 50000 --population-total-frame-budget 500000 --history population_run1.jsonl --study-db population_run1.db --checkpoint-dir population_run1_ckpts
```

### 2.1.5 `resume` 是什么

`resume` 的意思是“继续某一个 trial 的训练状态”。

它的概念是：

- 从某个 snapshot 恢复网络、优化器、buffer、epsilon、lineage 等状态
- 继续原来的训练链

它的用途是：

- 某个有潜力的 trial 被中断后继续跑
- 想把一次训练从较小预算续到更大预算
- 给 population 继承机制提供基础能力

它不是“继续整个 Optuna 搜索”，而是“继续某一个 trial”。

推荐命令：

```powershell
python main.py --resume run1_ckpts\snapshots\snapshot_0_20000.pt --mode normal --max-trial-frames 1000000 --history resume_run1.jsonl --checkpoint-dir resume_run1_ckpts
```

前提说明：

- 这条命令依赖 `run1_ckpts\snapshots\snapshot_0_20000.pt` 已经存在
- 如果你的 snapshot 文件名不同，只需要改这一处路径

### 2.1.6 `recheck-topk` 是什么

`recheck-topk` 的意思是“复核历史里最好的前 K 个配置”。

它的概念是：

- 重新用多个独立 seed 跑这些历史优质配置
- 检查它们是不是只在某一个 seed 下偶然跑得好

它的用途是：

- 排除偶然成功
- 检查配置是否真的稳定
- 为最终确认提供候选名单

简单说，它回答的问题是：

- “这个配置是真的强，还是运气好？”

推荐命令：

```powershell
python main.py --recheck-topk 3 --mode normal --history tpe_run1.jsonl --study-db tpe_run1.db
```

### 2.1.7 `final-confirm` 是什么

`final-confirm` 的意思是“最终验收”。

它的概念是：

- 对一个给定配置做固定 5-seed 的最终评测
- 用更严格的整体成功标准判断它是否可以被认定为稳定方案

它的用途是：

- 对候选最佳配置做最终拍板
- 把“看起来不错”升级为“可以认定稳定达标”
- 给需求目标“稳定达到 1000 分”提供最终证据

简单说，它回答的问题是：

- “这个配置能不能正式宣布通过？”

推荐命令：

```powershell
python main.py --final-confirm best_config.json --mode normal --max-trial-frames 1000000 --checkpoint-dir final_confirm_ckpts
```

前提说明：

- 这条命令依赖 `best_config.json` 已经存在
- 它通常来自你手工保存的最佳配置，或后续实验整理结果

### 2.1.8 `matrix` 是什么

`matrix` 的意思是“按预定义实验表批量执行一组对照实验”。

它的概念是：

- 一次运行多组有明确目的的配置或策略
- 用来做结构对比、协议对比、搜索器对比、基线对比

它的用途是：

- 做消融实验
- 做版本对比
- 做协议验证
- 给需求文档和实验报告提供结构化证据

它不是为了“直接找到最强配置”，而是为了“回答某种研究问题”。

四种矩阵的具体含义：

- `baseline`
  用途：做基线对照。
  它主要比较固定 baseline、历史最佳配置文件、以及当前 baseline 路线。

- `structure`
  用途：做模型结构与探索机制对照。
  它主要比较 `mlp / dueling_mlp` 与 `epsilon_greedy / noisy_net` 的组合差异。

- `protocol`
  用途：做状态表示与奖励协议对照。
  它主要比较 `low_dim_v1 / v2 / v3` 与 `reward_v1_sparse / reward_v2_ratio / reward_v3_gap_shaping` 这些训练底座组合。

- `searcher`
  用途：做搜索器对照。
  它主要比较 `tpe_fresh`、`warmstart_tpe`、`population_async` 三种搜索方式本身。

最重要的理解是：

- `protocol / structure / baseline / searcher` 是 `--matrix` 的取值
- 它们不是新的 `--search-strategy`
- `--search-strategy` 仍然只有 `tpe_fresh / warmstart_tpe / population_async`

推荐命令：

```powershell
python main.py --matrix baseline --matrix-budget debug_matrix --mode debug
```

### 2.1.9 `render` 是什么

`render` 的意思是“把历史最佳 agent 直接演示出来”。

它的概念是：

- 从 history 中找到最佳 trial
- 读取它的 checkpoint
- 用 greedy policy 在 `pygame` 里实时播放

它的用途是：

- 肉眼验证 agent 行为
- 展示当前最佳结果
- 快速做 demo

它不参与搜索，也不改写训练结果。

推荐命令：

```powershell
python main.py --render --history tpe_run1.jsonl --render-episodes 1 --render-fps 60
```

前提说明：

- 这条命令依赖 `tpe_run1.jsonl` 中已经有可用的最佳 trial 记录
- 同时该记录对应的 `checkpoint_path` 必须真实存在

### 2.1.10 `report` 是什么

`report` 不是独立任务，而是“附加输出开关”。

它的概念是：

- 在 baseline、search、recheck 等流程之后
- 额外生成 Markdown/JSON 报告文件

它的用途是：

- 把实验结果整理成可复盘、可对比、可归档的产物
- 降低后续人工翻 JSONL 的成本

推荐命令：

```powershell
python main.py --mode normal --max-trials 30 --history report_run1.jsonl --study-db report_run1.db --checkpoint-dir report_run1_ckpts --report
```

## 2.2 一张图理解整套流程

如果按最常见的使用顺序，这套 CLI 大致对应下面这个流程：

1. `baseline-only`
用途：先确认系统可运行

4. `recheck-topk`

用途：对候选最佳配置做最终验收

6. `render`
用途：把历史最佳 agent 直观展示出来

7. `report`
用途：把结果整理成总结文件

## 4. 启动命令的组成规则

### 4.1 通用模板

绝大多数命令都可以按下面这个结构理解：

```powershell
python main.py [任务类型] [运行模式] [搜索或训练预算] [输出位置] [可选覆盖参数] [附加功能]
```

最常见的标准搜索模板：

```powershell
python main.py --mode <debug|normal|deep> --max-trials <N> [其他可选参数]
```

例如：

```powershell
python main.py --mode normal --max-trials 30 --history normal_run1.jsonl --study-db normal_run1.db
```

这条命令的组成是：

- `python main.py`：运行主入口
- `--mode normal`：使用 `normal` 预算档位
- `--max-trials 30`：最多跑 30 个 trial
- `--history normal_run1.jsonl`：把历史记录写入该 JSONL
- `--study-db normal_run1.db`：把 Optuna study 写入该 SQLite 文件

### 4.2 先判断你要启动哪种任务

在拼命令之前，先确认这次要做的是哪一类任务：

1. `baseline-only`

只跑一次固定基线，不做搜索。

```powershell
python main.py --baseline-only --mode <debug|normal|deep>
```

2. `search`

正式超参搜索。默认就是这类。

```powershell
python main.py --mode <debug|normal|deep> --max-trials <N>
```

3. `resume`

从已有 V3 snapshot 继续同一个 trial。

```powershell
python main.py --resume <snapshot.pt> --mode <debug|normal|deep>
```

4. `recheck-topk`

对历史里最好的前 K 个配置做复核。

```powershell
python main.py --recheck-topk <K> --history <history.jsonl> --mode <debug|normal|deep>
```

5. `final-confirm`

对一个配置 JSON 做 5-seed 最终验收。

```powershell
python main.py --final-confirm <config.json> --mode <debug|normal|deep>
```

6. `matrix`

跑预定义实验矩阵。

```powershell
python main.py --matrix <baseline|structure|protocol|searcher> --matrix-budget <debug_matrix|normal_matrix> --mode <debug|normal|deep>
```

7. `render`

不训练、不搜索，直接渲染历史最佳 agent。

```powershell
python main.py --render --history <history.jsonl>
```

### 4.3 推荐拼接顺序

建议按下面顺序写命令，最不容易乱：

```powershell
python main.py
[任务类型]
--mode ...
--search-strategy ...
--max-trials ...
--max-trial-frames ...
--history ...
--study-db ...
--checkpoint-dir ...
[覆盖参数]
[附加功能]
```

也就是先写“做什么”，再写“预算多大”，再写“结果保存到哪”，最后再写“额外开关”。

## 5. 运行模式与预算

`--mode` 决定默认预算档位：

| 模式 | `max_trial_frames` 默认值 | `eval_interval_frames` | `eval_episodes` |
|---|---:|---:|---:|
| `debug` | 100000 | 10000 | 3 |
| `normal` | 1000000 | 20000 | 5 |
| `deep` | 5000000 | 50000 | 20 |

说明：

- `--max-trial-frames` 可以覆盖单个 trial 的最大训练帧数。
- `debug` 只适合验证流程，不适合判断是否能成功收敛到 1000 分。
- `normal` 是最常用的正式搜索档位。
- `deep` 用于更大预算的正式搜索和最终攻关。

```powershell
python main.py --mode normal --search-strategy warmstart_tpe --max-trials 30 --history warmstart_run1.jsonl --study-db warmstart_run1.db --checkpoint-dir warmstart_ckpts
```

### 6.5 population_async 自进化搜索

```powershell
python main.py --mode normal --search-strategy population_async --population-size 4 --population-eval-interval 20000 --population-exploit-interval 50000 --population-total-frame-budget 3000000 --history pop_run1.jsonl --study-db pop_run1.db --checkpoint-dir pop_ckpts
```

说明：

- 这是 V3 的自进化人口搜索入口
- 它主要按总帧预算推进，核心预算是 `--population-total-frame-budget`
- `--max-trials` 对 `population_async` 不是主要控制项

适用场景：

- 已经有一定历史积累
- 想验证“继承训练 + 在线变异”是否更高效
- 想推进 V3 自进化搜索主线

### 6.6 从 snapshot 继续训练

```powershell
python main.py --resume checkpoints\snapshots\snapshot_0_20000.pt --mode normal --max-trial-frames 1000000 --history resume_run1.jsonl --checkpoint-dir resume_ckpts
```

说明：

- `--resume` 继续的是“同一个 trial 的训练”
- 不是继续 Optuna 搜索，也不是恢复整个 study 调度器
- 输入必须是 V3 snapshot 文件路径

适用场景：

- 某个 trial 有潜力但预算不够
- 训练中断后要继续跑

```powershell
python main.py --recheck-topk 3 --mode normal --history normal_run1.jsonl
```

说明：

- 会从 `history` 中取当前最优的前 `K` 个配置
- 重新按固定复核 seed 跑一遍
- 结果会追加写回同一个 `history`

适用场景：

- 想确认好成绩不是偶然 seed
- 想筛选进入最终确认的候选配置

### 6.8 最终 5-seed 验收

```powershell
python main.py --final-confirm best_config.json --mode normal --max-trial-frames 1000000 --checkpoint-dir final_confirm_ckpts
```

说明：

- 输入是一个 JSON 配置文件
- 程序会固定用 5 个 seed 跑最终验收
- 每个 seed 会做强制最终评估
- 结果直接打印为 JSON

适用场景：

- 已经有候选最佳配置
- 想正式判断它是否能作为稳定达标方案

### 6.9 运行预定义实验矩阵

```powershell
python main.py --matrix baseline --matrix-budget debug_matrix --mode debug
```

可选矩阵：

- `baseline`
- `structure`
- `protocol`
- `searcher`

这些矩阵分别表示：

- `baseline`
  用来做基线与已有参考配置对照。

- `structure`
  用来判断“网络结构 / 探索头”是不是当前瓶颈。

- `protocol`
  用来判断“状态表示 / 奖励协议”这条训练底座是不是太弱。

- `searcher`
  用来判断“搜索器本身”是不是影响结果的主要原因。

可选预算档位：

- `debug_matrix`
- `normal_matrix`

适用场景：

- 做对照实验
- 做协议/结构/搜索器消融分析

### 6.10 动态画面演示

```powershell
python main.py --render --history search_history.jsonl
```

指定演示局数和帧率：

```powershell
python main.py --render --history search_history.jsonl --render-episodes 3 --render-fps 60
```

适用场景：

- 看当前最佳 agent 的实际行为
- 做演示或展示

## 7. 搜索策略说明

### 7.1 `tpe_fresh`

默认搜索策略。

特点：

- 从空白 TPE 搜索开始
- 会先跑 baseline
- 再按 Optuna/TPE 继续试验

### 7.2 `warmstart_tpe`

在 TPE 之前，把历史优质配置作为热启动样本。

特点：

- 适合你已经积累了一批历史搜索结果
- 能减少纯冷启动的时间浪费
- 仍然保持 TPE 的整体框架

### 7.3 `population_async`

V3 的自进化搜索策略。

特点：

- 多 worker 并行式地分块训练
- 周期性进行 exploit/explore
- 依赖 snapshot 进行继承和继续训练
- 会写入 `population_event` 记录

| 参数 | 默认值 | 说明 |
|---|---:|---|
| `--population-size` | `4` | 人口规模 |
| `--population-eval-interval` | `20000` | 每个 worker 的训练块大小 |
| `--population-exploit-interval` | `50000` | 全局 exploit/explore 周期 |
| `--population-total-frame-budget` | `None` | 整个人口搜索总帧预算 |

### 8.6 训练机制覆盖参数

这些参数是“强制覆盖项”。你传了以后，会覆盖 baseline 或 trial 内对应配置。

| 参数 | 默认行为 | 说明 |
|---|---|---|
| `--n-step` | 不覆盖 | 强制设为 `1 / 3 / 5` |
| `--priority` | 不覆盖 | 强制开启 PER |
| `--per-alpha` | 不覆盖 | 覆盖 PER alpha |
| `--per-beta-start` | 不覆盖 | 覆盖 PER beta 起点 |
| `--per-beta-train-updates` | 不覆盖 | 覆盖 beta 调度长度，按 `train_updates` 推进 |
| `--death-ratio` | 不覆盖 | 覆盖死亡惩罚比例 |
| `--alive-ratio` | 不覆盖 | 覆盖存活奖励比例 |
| `--reward-scale` | 不覆盖 | 覆盖 reward 缩放系数 |
| `--reward-clip` | 不覆盖 | 覆盖 reward 裁剪阈值 |

注意：

- `--priority` 只有“强制开启”，没有 `--no-priority`
- 没传的字段仍然由 baseline 默认值或搜索空间决定
- Optuna 日志里显示的原始采样参数，不一定等于最终训练配置
- 以 `search_history.jsonl` 里每条记录的 `config` 字段为准

## 13. `--final-confirm` 输出字段说明

`--final-confirm` 会直接向终端打印一个 JSON 对象，常见字段如下：

| 字段 | 含义 |
|---|---|
| `config` | 被验收的配置 |
| `seeds_used` | 本次使用的 5 个 seed |
| `all_eval_scores` | 所有 seed、所有最终评估 episode 的总分列表 |
| `per_seed` | 每个 seed 的详细 trial 结果 |
| `overall_success_rate_1000` | 所有最终评估 episode 中 `>=1000` 的比例 |
| `overall_median_score` | 所有最终评估分数的总体中位数 |
| `status` | `confirmed / not_confirmed / invalid` |
| `reason` | 若 `status=invalid`，这里给出原因 |

当前判定规则：

- `overall_success_rate_1000 >= 0.80`
- 且 `overall_median_score >= 1000`

同时满足时，`status = confirmed`。

## 15. 常见问题

### 15.1 `unrecognized arguments`

先执行：

```powershell
python main.py -h
```

当前帮助输出中应至少能看到这些 V3 参数：

- `--search-strategy`
- `--resume`
- `--recheck-topk`
- `--final-confirm`
- `--matrix`
- `--population-size`

如果没有，说明你当前运行的不是最新的 [main.py](D:/Python/强化学习agent/flappy_bird/main.py)。

### 15.2 Optuna 日志里显示的参数和我传入的不一样

这是正常的。

原因：

- Optuna 打印的是“采样参数”
- CLI 覆盖参数会在后处理阶段覆盖最终配置

所以最终以 `search_history.jsonl` 中每条记录的 `config` 字段为准。

### 15.3 `--render` 报 checkpoint 不存在

说明历史里虽然有最佳 trial 记录，但它对应的 `checkpoint_path` 不可用。常见原因：

- 你移动或删除了 checkpoint 文件
- 历史文件和 checkpoint 目录不是同一轮实验留下的
- 使用了旧 history，但没保留对应 checkpoint

### 15.4 怎样继续上一次搜索

如果你要继续同一个 Optuna 搜索，最简单的做法是重复使用同一组输出文件：

```powershell
python main.py --mode normal --max-trials 30 --history run1.jsonl --study-db run1.db --checkpoint-dir run1_ckpts
```

中断后再执行同样的命令即可。

如果你要继续“某一个 trial 的训练状态”，则使用：

```powershell
python main.py --resume run1_ckpts\snapshots\snapshot_0_20000.pt --mode normal --history run1_resume.jsonl --checkpoint-dir run1_resume_ckpts
```

这两者不是一回事：

- 重复同一个 `study-db`：继续整个搜索流程
- `--resume <snapshot>`：继续某个 trial 的训练状态

## 15.5 搜索结果很差时怎么排查

如果你跑完一轮标准搜索，出现下面这种情况：

- `Success: 0`
- `Top 1` 的 `median_score` 很低
- `Top 1` 的 `success_rate_1000 = 0`

不要立刻重复同样的 30 次搜索。更合理的是按下面顺序排查。

### 排查思路总览

先回答四个问题：

1. 系统本身能不能正常训练和评估
2. 当前训练底座是不是太弱
3. 当前模型结构是不是瓶颈
4. 当前搜索器是不是主要问题

### 第一步：先确认不是运行环境或流程问题

推荐命令：

```powershell
python main.py --baseline-only --mode debug --history diag_baseline.jsonl --checkpoint-dir diag_baseline_ckpts
```

要看什么：

- 能不能正常跑完
- 会不会报错退出
- `history`、`checkpoint`、`snapshot` 是否正常生成

如果这一步都异常，先不要继续正式搜索。

### 第二步：先查训练底座是否过弱

推荐命令：

```powershell
python main.py --matrix protocol --matrix-budget debug_matrix --mode debug
```

这一步回答的问题是：

- `low_dim_v1 / v2 / v3` 哪种状态表示更好
- `reward_v1_sparse / reward_v2_ratio / reward_v3_gap_shaping` 哪种奖励协议更有学习信号

如果这一步里所有组合都很差，通常说明：

- 训练底座本身还不够强
- 问题优先不在搜索器，而在状态表示、奖励设计或训练协议

### 第三步：再查模型结构是不是瓶颈

推荐命令：

```powershell
python main.py --matrix structure --matrix-budget debug_matrix --mode debug
```

这一步回答的问题是：

- `mlp` 和 `dueling_mlp` 哪个更合适
- `epsilon_greedy` 和 `noisy_net` 哪个更适合当前问题

如果 `protocol` 有明显差异、但 `structure` 差异不大，通常说明：

- 主要瓶颈在训练协议
- 不是网络结构问题

### 第四步：如果底座没问题，再比较搜索器

推荐命令：

```powershell
python main.py --matrix searcher --matrix-budget debug_matrix --mode debug
```

这一步回答的问题是：

- `tpe_fresh`
- `warmstart_tpe`
- `population_async`

这三种搜索方式，哪一种在当前底座上更容易找到高分配置。

如果 `protocol` 和 `structure` 里已经出现明显更好的方向，而 `searcher` 里 `warmstart_tpe` 或 `population_async` 更好，那么后续主线就不应该继续死跑默认 `tpe_fresh`。

### 第五步：再做下一轮正式搜索

如果前面排查表明：

- 某个 `protocol` 方向更好
- 某个 `structure` 方向更好
- 某个 `searcher` 方向更好

那么下一轮正式搜索应优先沿那个方向继续，而不是完全重复旧命令。

例如：

- 如果怀疑搜索器冷启动太弱，优先用 `warmstart_tpe`
- 如果怀疑继承训练更有效，优先用 `population_async`

### 一套最推荐的排查顺序

如果你想要最省事的一套顺序，可以直接按下面四条命令跑：

1. 基线检查

```powershell
python main.py --baseline-only --mode debug --history diag_baseline.jsonl --checkpoint-dir diag_baseline_ckpts
```

2. 协议排查

```powershell
python main.py --matrix protocol --matrix-budget debug_matrix --mode debug
```

3. 结构排查

```powershell
python main.py --matrix structure --matrix-budget debug_matrix --mode debug
```

4. 搜索器排查

```powershell
python main.py --matrix searcher --matrix-budget debug_matrix --mode debug
```

### 什么时候不建议直接继续跑 30 次标准搜索

下面这些情况下，不建议立刻再跑一轮完全相同的 `--mode normal --max-trials 30`：

- `Success: 0`
- 最好 trial 的 `median_score` 还很低
- `success_rate_1000` 全部为 `0`
- 你还没有做过 `protocol` 和 `structure` 的排查

因为这通常意味着：

- 不是“试得不够多”这么简单
- 更可能是“训练底座方向还没选对”

## 17. 每种指令可能得到的结果格式

这一节专门说明“命令执行后你会看到什么格式的结果”，避免把“程序跑完了”和“模型已经成功了”混为一谈。

### 17.1 `--baseline-only` 的结果格式

典型终端输出结构：

```text
[BOOT] Loading runtime...
[MODE] debug  |  max_trial_frames=100000
[BASELINE-ONLY] Running single baseline trial...
...进度条...

[SUMMARY]
Trials: 1  Success: 0  Failure: 1
Best trial: -1  Objective: 30000.0
Top 1: trial=-1  obj=30000.0  median=0.0  sr=0.0
```

说明：

- `baseline-only` 通常不会打印 `SUCCESS Trial #...` / `FAILED Trial #...` 这种 TPE trial 结尾行
- 它的最终结果主要看 `[SUMMARY]`
- 详细字段写入 `history` 的 `trial` 记录中

### 17.2 标准搜索 `tpe_fresh` / `warmstart_tpe` 的结果格式

典型终端输出结构：

```text
[BOOT] Loading runtime...
[MODE] normal  |  max_trial_frames=1000000

[STAGE 0] Baseline verification (independent, not counted in max_trials)...
[STAGE 0] Baseline complete. status=failure

[INFO] Study: 0 Optuna trials completed, 30 remaining
[INFO] Max trial frames: 1000000

==================================================
Trial #0  |  Source: TPE  |  Seed: 11
Config: lr=1.00e-04, gamma=0.9900, ...
==================================================
...进度条...
SUCCESS Trial #0  train_frames=640000  median=1000.0  sr=70.00%  obj=640000.0
```

失败时常见格式：

```text
FAILED Trial #3  reason=max_frames_reached  best_eval=420.0  obj=30000.0
```

最后会统一输出：

```text
[SUMMARY]
Trials: 31  Success: 1  Failure: 30
Best trial: 0  Objective: 640000.0
Top 1: trial=0  obj=640000.0  median=1000.0  sr=0.7
Top 2: trial=7  obj=...  median=...  sr=...
```

说明：

- `warmstart_tpe` 和 `tpe_fresh` 的终端结果格式基本相同
- 差别在于搜索初始化方式，不在于输出结构

### 17.3 `population_async` 的结果格式

典型终端输出结构：

```text
[BOOT] Loading runtime...
[MODE] normal  |  max_trial_frames=1000000

[POPULATION] 2 workers, budget=500000 frames
...每个 worker 分块训练与评估进度条...

[SUMMARY]
Trials: 18  Success: 0  Failure: 18
Best trial: 10002  Objective: 28000.0
Top 1: trial=10002  obj=28000.0  median=520.0  sr=0.0
```

说明：

- `population_async` 终端里不会像 TPE 那样逐个打印 `Trial #... Source: TPE`
- 它的详细过程主要写入 `history`
- 除了普通 `trial` 记录，还会写入 `record_type = "population_event"` 的事件记录

### 17.4 `--resume` 的结果格式

典型终端输出结构：

```text
[BOOT] Loading runtime...
[MODE] normal  |  max_trial_frames=1000000
...进度条...

[SUMMARY]
Trials: 1  Success: 1  Failure: 0
Best trial: 0  Objective: 620000.0
Top 1: trial=0  obj=620000.0  median=1000.0  sr=0.7
```

说明：

- `--resume` 继续的是某个 snapshot 对应的同一条训练链
- 详细字段主要看 history 中该条 `trial` 记录的 `trial_type = "resume"`
- 详细字段主要看 history 中该条 `trial` 记录的 `parent_snapshot_ref`
- 详细字段主要看 history 中该条 `trial` 记录的 `lineage_*`

### 17.5 `--recheck-topk` 的结果格式

典型终端输出结构：

```text
[BOOT] Loading runtime...
[MODE] normal  |  max_trial_frames=1000000
[RECHECK] Rechecked top 3 configs.
```

说明：

- 终端只会给一个简短完成提示
- 详细结果不会完整打印在屏幕上
- 详细数据会追加写入 `history`，记录类型为 `record_type = "recheck"`
- 如果同时加了 `--report`，还会生成 `recheck_summary.json`

### 17.6 `--final-confirm` 的结果格式

这条命令会直接打印一个 JSON 对象。

典型格式：

```json
{
  "config": { "...": "..." },
  "seeds_used": [1001, 1002, 1003, 1004, 1005],
  "all_eval_scores": [1000, 1032, 980, 1004],
  "per_seed": [
    {
      "trial_id": 99991001,
      "status": "success",
      "median_score": 1000.0,
      "success_rate_1000": 0.75
    }
  ],
  "overall_success_rate_1000": 0.82,
  "overall_median_score": 1005.0,
  "status": "confirmed"
}
```

可能出现的 `status`：

- `confirmed`
- `not_confirmed`
- `invalid`

### 17.7 `--matrix` 的结果格式

这条命令会打印一个很短的 JSON 摘要。

典型格式：

```json
{
  "matrix": "baseline",
  "budget": "debug_matrix",
  "result_count": 3
}
```

说明：

- `result_count` 表示本次矩阵执行得到的结果条目数
- 它不等于“成功次数”
- 若某个矩阵项缺少配置文件，内部结果可能是 `status = "skipped"`，但终端只打印汇总 JSON

### 17.8 `--render` 的结果格式

典型终端输出结构：

```text
[BOOT] Loading runtime...
[RENDER] Loading best trial from search_history.jsonl
```

之后会弹出 `pygame` 窗口。

如果失败，通常直接抛错退出，例如：

```text
[FATAL] Checkpoint file not found: ...
```

### 17.9 `--report` 的结果格式

`--report` 本身不是独立任务，而是附加开关。

它的效果不是在终端打印一大段结果，而是生成报告文件：

- `summary_report.md`
- `topk_summary.json`
- `recheck_summary.json`
- `experiment_manifest.json`

## 18. 成功数据的标识规则

这一节只回答一个问题：什么才算“真的成功”。

### 18.1 trial 成功的最直接标识

对普通训练记录来说，最直接的成功标识是：

- 终端出现 `SUCCESS Trial #...`
- 或 `history` 中该条记录满足 `record_type = "trial"` 且 `status = "success"`

这类数据表示该 trial 已经达到项目定义的“稳定成功”门槛。

### 18.2 trial 成功的判定标准

当前代码里，普通 trial 的稳定成功判定是：

- `median_score >= 1000`
- 且 `success_rate_1000 >= 0.70`

也就是最终评估中：

- 中位数至少 `1000`
- 且至少 `70%` 的评估局分数 `>=1000`

只有满足这两个条件，`status` 才会被写成 `success`。

### 18.3 “跑完了”不等于“成功了”

下面这些情况都属于“程序执行完成了，但结果不算成功”：

- `status = "failure"`
- `failure_reason = "max_frames_reached"`
- `[SUMMARY]` 里 `Success: 0`
- `Top 1` 的 `median < 1000`
- `Top 1` 的 `sr < 0.70`

这类数据仍然是有效实验数据，可以用于比较、warmstart、人口继承或继续调参，但不能当作“已找到成功配置”。

### 18.4 `final-confirm` 的成功标识

`final-confirm` 的成功标识更严格。

只有当输出 JSON 中：

- `status = "confirmed"`

才表示它通过了最终验收。

当前最终验收规则是：

- `overall_success_rate_1000 >= 0.80`
- 且 `overall_median_score >= 1000`

也就是：

- 全部最终评估样本里，至少 `80%` 分数 `>=1000`
- 且总体中位数至少 `1000`

如果只是：

- `status = "not_confirmed"`

说明这个配置“有一定效果”，但还不能认定为最终稳定达标方案。

### 18.5 `recheck` 的成功标识

复核记录里，最重要的成功字段是：

- `recheck_passed = true`

它表示这条历史优质配置在复核 seeds 下整体通过。

辅助判断字段包括：

- `recheck_median`
- `recheck_success_rate`
- `failed_seeds`

如果：

- `recheck_passed = false`

说明该配置可能存在 seed 偶然性，不应直接当作稳定最终答案。

### 18.6 `population_async` 中什么数据算成功

`population_async` 里真正的“成功数据”仍然是普通 `trial` 记录，而不是 `population_event`。

判断方法和普通 trial 一样：

- `record_type = "trial"`
- `status = "success"`

不要把下面这些误认为成功：

- `record_type = "population_event"`
- `Best trial` 只是暂时最优，但 `status` 不是 `success`
- `objective` 很小，但 `median_score` 仍未达到 `1000`

### 18.7 终端里最容易识别的成功信号

如果你只想快速看终端，不想立刻翻 JSONL，可以按下面顺序判断：

1. 看有没有 `SUCCESS Trial #...`
2. 看 `[SUMMARY]` 里的 `Success` 是否大于 `0`
3. 看 `Top 1` 的 `median` 是否 `>= 1000`
4. 看 `Top 1` 的 `sr` 是否 `>= 0.70`
5. 如果是 `--final-confirm`，只认 `status = "confirmed"`

### 18.8 最推荐的人工检查顺序

最稳的检查顺序是：

1. 先看终端是否报错退出
2. 再看 `[SUMMARY]` 或 JSON 输出
3. 再看 `search_history.jsonl` 对应记录的 `status`
4. 再看 `search_history.jsonl` 对应记录的 `median_score`
5. 再看 `search_history.jsonl` 对应记录的 `success_rate_1000`
6. 如果要确认是否真的稳定，再看 `recheck_passed`
7. 如果是最终验收，再看 `final-confirm` 的 `status = "confirmed"`
